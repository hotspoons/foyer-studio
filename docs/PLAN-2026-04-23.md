# Foyer Studio — Plan, 2026-04-23

Forward-looking backlog assembled from `docs/PLAN.md` notes + Rich's
2026-04-22 asks (double-click track editor, sequencer chip on strips,
start automation) + the remaining slices from the 2026-04-22 session-
lifecycle push. Items are grouped by theme and ordered roughly by
(user-visible leverage × ease). Anything already shipped and struck
through in `PLAN.md` is not repeated here.

Legend:

- **[small]** — under a day, bounded surface area
- **[medium]** — multi-file vertical slice, half-done to 1–2 days
- **[large]** — new subsystem / architecture, multi-day
- **[blocking]** — something else waits on this
- *(path)* — single-word cue for where the work starts

## Top of the list — priorities from Rich, 2026-04-22

### A. Track editor ergonomics **[small]** *(track-strip)*

Double-clicking a track opens the track editor today on the mixer
strip via `@dblclick=${() => this._startRename()}` — but Rich wants
the full editor, and the timeline lane-head has no double-click
binding at all.

Work:

- Timeline `.lane-head` gets `@dblclick=${() => openTrackEditor(track.id)}`.
- Mixer strip: double-click behaviour becomes "open editor"; rename
  moves to a dedicated button or stays on a second affordance
  (e.g. Enter on the name with a single-click focus). Keep the
  existing name-inline-edit accessible — just don't make it the
  primary double-click target anymore.

### B. MIDI track context menu: above-the-fold editor access **[small]** *(timeline-view)*

Today the lane-head context menu on MIDI tracks stacks Track editor →
Open piano roll → Open beat sequencer → MIDI patches. Rich wants the
MIDI-specific items promoted above the fold so three-click access
(right-click → read → click) reaches the piano roll / sequencer
without scanning.

Work:

- Reorder `_onLaneHeadContext` items: Open piano roll / Open beat
  sequencer come first on MIDI tracks, Track editor below, MIDI
  patches next, separator, then generic track actions.

### C. "Sequencer-driven" strip marker **[small]** *(track-strip, mixer)*

A MIDI track backing a sequencer-owned region is still labelled just
`midi`. Mark it so the engineer spots which tracks are driven by the
beat sequencer at a glance — e.g. `midi(seq)` or a small chip.

Work:

- Compute "has any region with an active `foyer_sequencer` layout"
  per track. Source it from `store.state.session.tracks[i]` joined
  with `_regionsByTrack` — cheap, already in memory.
- Render as a second chip (`SEQ`) next to the kind label on both
  the mixer strip and the timeline lane-kind text.

### D. Automation **[large]** *(new subsystem)*

Schema types already exist (`AutomationLane`, `AutomationPoint`,
`AutomationMode` in [value.rs][val]). Nothing else: no shim read,
no shim write commands, no UI. This is the big next initiative.

Phased plan:

1. **A — read-only display** **[medium, blocking]**
   - Shim: walk `Stripable::automation_control_iter()` per route,
     fill an `AutomationLane` per control with current
     `AutomationList::AutoState` + sampled points. Emit inside the
     track snapshot (new field: `Track.automation_lanes`).
   - Frontend: `<foyer-automation-lane>` under track / region, paints
     a polyline from the point list + a live dot at the playhead
     driven by ControlUpdate of the underlying parameter.
   - Lane visibility toggle per track (auto-panel-style: a "show
     automation" affordance that expands beneath the lane).

2. **B — manual edit** **[medium]**
   - New commands: `SetAutomationMode { lane_id, mode }`,
     `AddAutomationPoint { lane_id, point }`,
     `UpdateAutomationPoint`, `DeleteAutomationPoint`,
     `ReplaceAutomationLane { lane_id, points }`.
   - Shim: map to `AutomationList::add`, `erase`, `rt_add`, wrap
     in Session commands for undo grouping.
   - Frontend: click on the lane strip to add a point, drag to
     move, delete to remove. Curve editor with linear / discrete /
     ramp variants (track Ardour's `Evoral::ControlEvent` shape).

3. **C — record-mode automation** **[medium]**
   - Mode dropdown per lane: Off / Manual / Play / Write / Touch /
     Latch (already in schema).
   - In Write/Touch, the user's fader movements get recorded into
     the lane while transport rolls. UI shows a red "writing"
     indicator and a playback badge.

4. **D — integration polish** **[small]**
   - Gain / pan / mute lanes on every track as default-visible
     lanes (Ardour behaviour).
   - Plugin parameter lanes in the plugin panel.
   - Automation moves / survives region operations (split, move,
     duplicate) — inherits the Ardour Playlist semantics for free
     if we attach lanes at the right level.

Decision gate to log when phase A lands: whether the lane UI lives
under each track in the timeline (like Ardour), as a separate Foyer
tile ("automation view"), or both. Two surfaces is probably right —
inline for quick glance, tile for bulk editing.

## Session lifecycle — remaining **[medium]** *(sessions, orphans)*

Carried from the 2026-04-22 push. Feature user-visible, edges still
sharp.

- **"Already open by path" short-circuit** **[small]**
  `Command::LaunchProject` should call
  `SessionRegistry::find_by_path(canonical_path)` before spawning and
  `SelectSession` to the existing id when it matches. Drops the
  two-backends-same-project confusion the shim already guards
  against via lock files.

- **Orphan reattach (live shim, lost sidecar)** **[medium]**
  `ReattachOrphan` handler in [ws.rs][ws] still returns
  `reattach_unimplemented`. Wire it by exposing a
  `CliSpawner::attach(socket_path)` that skips
  `launch_and_wait_for_shim` and goes straight to
  `HostBackend::connect(socket)`. Then `swap_backend` pulls in the
  existing shim as a full session.

- **Session → Recent submenu** **[small]**
  Recents only surface from the welcome screen. Add a Session →
  Recent cascade in `main-menu.js` (the `[action-id="session.recent"]`
  pattern) so opening a recent doesn't require closing the active
  session first.

- **New-session creation** **[medium]**
  Welcome screen's "New session…" is stubbed. Needs
  `Command::CreateEmptySession { path, template?, sample_rate? }` in
  schema → sidecar → shim. Shim side: `Session::new(path)` with a
  minimal XML template. Template picker is phase 2.

- **Per-connection session selection** **[small, not blocking]**
  `Command::SelectSession` sets a sidecar-wide focus today. For
  multi-browser / multi-monitor pop-outs, each WS connection should
  track its own `current_session_id`. Mechanical refactor of
  `dispatch_command` to carry a per-connection state ptr.

## Audio — ingress, high-rate, jitter buffer **[medium-large]**

Three separable items; all carried from earlier notes.

- **Browser → DAW ingress (soft input port)** **[large]** — Decision
  24 has the sketch. `AudioIngressOpen` is wired to a "not
  implemented" error today. Shim-side class parallel to `MasterTap`,
  registers a soft input via `PortManager::register_input_port()`,
  feeds buffers from the IPC drain. ~200–400 lines of RT-disciplined
  Ardour C++. Latency calibration
  (`Backend::measure_latency()`) should land alongside.

- **96 / 192 kHz remote** **[medium]** — shim delivers at Ardour's
  session rate, no resampling. For true high-rate browsers, either
  set Ardour's session rate to match (user configurable in Ardour
  directly, not Foyer's problem) or land a resampler in the shim —
  `speex-resampler` or `rubato` (already a dep). Needs audio config
  UI on the client (codec pick, sample-rate pick).

- **Jitter buffer redesign** **[medium]** — current scheduling drives
  BufferSource starts straight from the decode callback. Move
  scheduling into a dedicated 10 ms interval tick with a
  `PriorityQueue<AudioBuffer>` keyed by decode timestamp. Drains at
  steady rate regardless of WebSocket packet burstiness. Kill the
  "resetting playhead" 300 ms pops. Pre-requisite for publishing
  the remote-mix feature as anything beyond "scratch-take grade."

## Mixer / routing / buses **[medium]**

- **Bus editing UI** — bus data model is already populated by the
  sidecar; no UI to route sends or create new buses. Needs a
  `<foyer-bus-panel>` surface with drag-targets on track strips.
  Schema primitives (`Send`, `Group`, bus assign in `TrackPatch`)
  already exist.

- **Multi-track add / delete** **[medium]** — schema has no
  `CreateTrack` / `DeleteTrack`. Add commands + shim handlers
  (`Session::new_audio_track`, `Session::new_midi_track`,
  `Session::remove_route`). UI: mixer/timeline context menu.

- **Track selection → batch ops** — delete/add/rename across a
  selection. Selection set already exists in the store; just the
  batch fan-out is missing.

- **Modifier-key resize-all** — hold a modifier while dragging a
  channel's width handle to resize every strip (already shipped for
  lanes in the timeline; wasn't ever done for the mixer's
  `channel-resize`).

- **Mousewheel over track labels → vertical scroll** — done for the
  timeline. Check the mixer behaves the same.

## Floating windows class **[medium]** *(window manager)*

Floating dialogs (MIDI editor, plugin editor, plugin config) should
live in a different class from core tiles. Wanted affordances:

- Group dock / undock as a whole
- Minimize all overlays to a single dock chip
- Separate move / restore cycle from the main tile layout

Pattern is an "overlay layer" parallel to the existing floating-
tiles layer but with group-scoped show/hide. About a day to design
+ a day to implement.

## MCP tools **[medium]**

Rich's long-standing ask. Two surfaces:

1. **Server-side** — expose Foyer control verbs (transport,
   track, region, MIDI edit, sequencer layout set, plugin preset
   apply, etc.) as MCP tools so an LLM agent can drive the DAW.
2. **Client-side** — agent panel calls the MCP tools; existing UI
   bindings become agent-accessible.
3. **Novel tools** — spectral analyzers, histograms, audio
   snippets, beat-sequencer helpers, MIDI-editing helpers. Each is
   a bounded tool definition on top of primitives that already
   exist in the schema.

Priority after automation lands, since automation is what turns a
passive agent into a useful mix assistant.

## Open design questions

- **Automation lane surface** — inline under the track in the
  timeline (like Ardour), standalone tile, or both? Pick when A1
  is ready to render.

- **Sequencer preview routing** — browser-synth (shipped) is the
  stop-gap. Real preview routes through the DAW. Does the user
  want "preview-only" that doesn't touch the region's note list,
  or a transparent "press key → add note to the sequencer"
  workflow? The latter is basically MIDI record mode and opens a
  record-to-sequencer can of worms.

- **Orphan reattach UX** — after a reattach succeeds, should the
  session fall into the switcher as the active view? Today the
  stubbed path just errors; the real path should auto-select.

- **New-session templates** — Ardour ships with a few built-in
  session templates. Mirror them in Foyer or start from "empty
  mono + stereo + master" and let the user build up?
